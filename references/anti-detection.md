# 反风控 / 拟人操作规范

> 所有"写"动作（打招呼 / 要简历）必须经过 `lib/rhythm.mjs` 节奏层。本文件是其设计依据。
> 平台风控盯的是**"非人类模式"**，不是单个动作。三个最危险的信号：节奏整齐、批量同质、脚本指纹。
> 核心心法：**优先"像真实点击"，而不是优先"最省实现代码"。**

## 一、节奏 / 批量（硬数值）

| 规则 | 默认值 | 说明 |
|------|--------|------|
| 单轮打招呼上限 | **3~5 人** | 绝不一次刷几十个 |
| 相邻"写"动作间隔 | **随机 25~70s** | 关键是**随机**，不能固定 |
| 每 N 人做一次校验 | **每 3 人** | 风控/状态轻量探针 |
| 每日封顶 | 按每个岗位每日目标人数 | 达标即停，不按时间机械刷 |
| 页面操作前留渲染时间 | 1500/1000/500ms | 进页/路由变/DOM 变 |

## 二、拟人化操作

- **点击优先级**：原生 CDP 点击（`opencli browser click`，含真实坐标）→ CDP 坐标点击 → DOM `.click()`（仅一次性兜底）。绝不默认 DOM click。
- **逐字输入**：`opencli browser type`（逐字符派发键盘事件），不用一次性 `fill`（瞬时填充像脚本）。
- **话术差异化**：模板留称呼/岗位变量，逐人轮换，不固定群发。
- **禁止脚本式定位**：不按 DOM 序号点"第 N 个按钮"；无唯一 id 时先 eval 给目标按钮打临时 `data-*` 标记，再用 CSS selector 点。

## 三、三次身份校验（防发错人）

发消息前三次确认当前线程身份 == 目标候选人：① 切线程后 ② 写入消息前 ③ 点发送前。
校验信号优先级：选中卡片 id → 右侧面板姓名 → 线程标题。任一不一致 → 立即停，不发。

## 四、打开候选人线程（独立动作，最多 3 次）

定位卡片（稳定 id）→ 滚动到可见中央 → 等列表稳定 → 点击 → 验证切换（面板/标题/输入区变化）。
最多 1 主 + 2 备用 = 3 次。仍失败即暂停，不递归乱点 / 坐标试错。

## 五、发送成功确认

不能只看"API 返回成功"。优先级：输入框清空 → 出现"送达"标记 → 打招呼场景按钮文案变化（如"打招呼"→"继续沟通"）。

## 六、风控信号检测 → 立即熔断

| 信号 | 应对 |
|------|------|
| 验证码 / 人机验证 / 异常验证 | 立即停整轮 |
| 登录态失效 / 被踢下线 | 停跑，提示重新登录 |
| 前 1~3 个成功、后续连续失败 | **优先判定当日沟通权益耗尽**（不是 selector 问题），进可恢复暂停态，保留进度 |
| 页面提示"操作频繁/访问异常" | 立即停整轮 |

> 关键经验：**"前几个成功后续全失败"第一反应不是怀疑代码，而是查权益耗尽。**

## 七、失败处理：一律进"可恢复暂停态"

暂停时只说 5 件事：① 卡在哪步 ② 已按最小快路径试过 ③ 最可能原因 ④ 用户最小要做什么 ⑤ 怎么恢复。不无脑重试。

## 八、一人一事务

处理完一个候选人完整闭环（定位→校验→打招呼→确认→记忆回写）再下一个。
禁止"先批量识别一批人，再回头按序号批量点按钮"——这是最典型的脚本模式，也最易点错人。

## 九、请求伪装

下载/接口请求带完整浏览器指纹头（UA + sec-ch-ua + sec-fetch-* + accept-language）；同源接口 `credentials:include`；
大文件（PDF）避免页面内 fetch 撞 CORS，走带 cookie 的服务端下载（本项目 `boss attachment` 即此方案，零弹窗）。
